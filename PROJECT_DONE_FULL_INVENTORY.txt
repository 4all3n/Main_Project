==================================================
1) BACKEND IMPLEMENTATION (FASTAPI + NLP + ML)
==================================================

1.1 API server (backend/app.py)
- FastAPI app initialized as MindfulMomentum Backend API.
- Endpoint POST /api/analyze-journal:
  - Accepts JSON body { text }.
  - Calls analyze_journal(text) from mindful_nlp.py.
  - Returns:
    - status
    - calculated_mood_score
    - overall_themes
    - paragraph_breakdown
- Endpoint GET /api/get-insight/{user_id}:
  - Calls generate_personalized_insight(user_id) from mindful_ml.py.
  - Returns status=success + data payload when valid.
  - Returns status=error + robust message when sparse/invalid data conditions occur.
- Uvicorn startup runner included for local serving on 0.0.0.0:8000.

1.2 NLP engine (backend/mindful_nlp.py)
- Uses NLTK + VADER sentiment analyzer.
- Paragraph segmentation from journal text.
- Sentence-level sentiment scoring to avoid dilution.
- Regex tokenizer for punctuation/noise-robust token extraction.
- POS-tag based theme extraction (nouns/adjectives).
- Stopword filtering + frequency ranking.
- Produces:
  - mapped mood score (1-5)
  - top overall themes
  - paragraph-level themes
- Handles empty/no-sentence input safely with defaults.

1.3 ML insight engine (backend/mindful_ml.py)
- Reads wellness, sleep, and Fitbit steps data per user.
- Data fusion by date across systems.
- Feature engineering:
  - sleep_score_yesterday
  - steps_yesterday
  - sleep_3d_avg
- Mood target categorization to binary class.
- Defensive checks:
  - not enough data days
  - no mood variance
- Model strategy:
  - RandomForestClassifier
  - Dynamic SMOTE based on class minimum
  - Imputation with mean strategy
  - Training + model serialization with joblib
- Reuse existing model if available for speed.
- Feature importance graph generation to output_graphs.
- API data payload includes:
  - user_id
  - top_feature
  - top_feature_impact_percent
  - insight_message
  - graphs_generated
  - data_days_used
  - latest_feature_values
  - feature_importances list

6) JOURNAL WORKFLOW (STORAGE, HISTORY, DETAIL NLP CACHE)
==================================================

6.1 Storage architecture
- mobile-app/lib/journal-storage.ts:
  - New canonical storage model: array of journal records with unique IDs.
  - Record fields: id, date, title(optional), content, createdAt, updatedAt, analysis cache.
  - Legacy migration from old map/object storage to array records.
  - Upsert/read/write helpers.
  - Source hash utility for NLP cache invalidation.

6.2 Create/Edit page
- mobile-app/app/(tabs)/journal/create.tsx:
  - Create and edit via record ID.
  - Optional title allowed (no title required to save).
  - Content required to save.
  - Saves as unique records; does not overwrite by date key anymore.
  - Preserves prior analysis cache only when source hash unchanged.
  - Invalidates cache when title/content changed.

6.3 Journal history page
- mobile-app/app/(tabs)/journal/index.tsx:
  - Loads all entries from array storage.
  - Sorts by latest update timestamp.
  - Displays all journals (not just one per date).
  - Handles untitled entries with fallback label.
  - Supports open/edit/delete by entry ID.

6.4 Journal detail page
- mobile-app/app/(tabs)/journal/[date].tsx:
  - Uses entry ID routing param.
  - Displays full journal content.
  - NLP analysis shown at bottom of detail page (not on create page).
  - Cached analysis retrieval from entry storage.
  - Re-fetches analysis only when source hash changed.
  - Persists fresh analysis back into entry cache with analyzedAt timestamp.
  - Graceful loading and error states.
